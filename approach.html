Phase 1: Discover

Goal: Understand what exists today and what the organization actually needs.

What I Do

  • Stakeholder interviews - Talk to SOC analysts, security engineers, IT ops, compliance team
  • Current state assessment - Document existing SIEM, log sources, detection rules
  • Pain point identification - What's not working? What's missing?
  • Requirements gathering - Compliance needs, use cases, integration requirements

Key Questions I Ask

Stakeholder Key Questions
SOC Analysts What alerts are useful? What's noise? What takes too long to investigate?
Security Engineers What's hard to maintain? What integrations are missing?
Compliance Team What reports are required? What retention periods? What audits?
IT Operations What log sources exist? What's the data volume? Network constraints?
Leadership What's the timeline? Budget? Success criteria?

Deliverable: Discovery Report

  • Current state documentation
  • Gap analysis
  • Risk assessment
  • Preliminary recommendations
ğŸ’¡ Interview Tip

"I always start with discovery because assumptions kill projects. I've seen migrations fail because someone assumed all detection rules were documented - they weren't. Discovery surfaces these risks early."

Phase 2: Design

Goal: Create the blueprint for the new SIEM environment.

What I Produce

  • High-Level Design (HLD) - Architecture overview, component selection, integration points
  • Low-Level Design (LLD) - Detailed configurations, naming conventions, RBAC model
  • Migration Plan - Phased approach, timeline, dependencies, rollback strategy
  • Test Plan - How we'll validate everything works

Design Decisions I Make

Decision Area Considerations
Platform Selection Splunk vs Sentinel vs others - based on requirements, budget, existing stack
Architecture Pattern Cloud, on-prem, hybrid - based on data residency, latency, cost
Log Source Priority Which sources first? Critical security value vs effort
Detection Strategy Migrate all rules? Use built-in content? Build custom?
Integration Approach SOAR, ticketing, threat intel - what connects to what?

Phase 3: Build

Goal: Set up the new SIEM environment and prepare for data.

What I Build

  • Infrastructure - Workspaces, clusters, forwarders, connectors
  • Ingestion pipelines - Data connectors, parsing rules, field mappings
  • Detection content - Translated rules, new analytics, correlation searches
  • Dashboards & workbooks - Operational views, investigation tools
  • Automation - Playbooks, response actions, integrations

Phase 4: Migrate

Goal: Move data sources and detection content to the new platform.

My Phased Approach

Phase What Migrates Why This Order
Wave 1 Authentication logs (AD, Entra ID) High security value, well-understood format
Wave 2 Endpoint logs (Windows, EDR) Critical for detection, largest volume
Wave 3 Network logs (Firewall, Proxy) Network visibility, perimeter detection
Wave 4 Cloud logs (AWS, Azure, GCP) Cloud-native threats, compliance
Wave 5 Application logs (O365, custom apps) Business context, insider threat

Phase 5: Validate

Goal: Prove the new SIEM works as well as (or better than) the old one.

What I Validate

  • Data completeness - All expected logs are arriving, no gaps
  • Parsing accuracy - Fields are extracted correctly
  • Detection parity - Alerts that fired before still fire now
  • Performance - Search times, dashboard load times, alert latency
  • Workflow testing - End-to-end investigation scenarios

Parallel Running

I always run both SIEMs in parallel for 2-4 weeks:

  • Same logs go to both platforms
  • Compare alerts side-by-side
  • Identify gaps before cutover
  • Build team confidence
âš ï¸ Common Mistake

Never skip parallel running. I've seen organizations cut over too early and miss critical alerts. The cost of running two SIEMs for a month is far less than missing a breach.

Phase 6: Handover

Goal: Transfer knowledge and ensure the team can operate independently.

What I Deliver

  • Operational runbooks - Day-to-day procedures, troubleshooting guides
  • Training sessions - Platform training, query writing, investigation workflows
  • Documentation - Architecture docs, configuration references, decision logs
  • Knowledge transfer - Shadowing, Q&A sessions, hands-on exercises
  • Support transition - Hypercare period, escalation paths

Success Criteria

Metric Target
Log source coverage 100% of planned sources ingesting
Detection parity All critical rules migrated and validated
Team readiness Team can operate without consultant support
Documentation complete All runbooks and guides delivered
Legacy decommissioned Old SIEM shut down (after retention period)

Why This Approach Works

  • Reduces risk - Phased approach catches issues early
  • Builds confidence - Stakeholders see progress throughout
  • Ensures adoption - Team is trained and ready before go-live
  • Creates documentation - Knowledge doesn't walk out the door with me
