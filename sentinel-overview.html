Microsoft Sentinel

Cloud-native SIEM and SOAR built on Azure

What is Microsoft Sentinel?

Microsoft Sentinel is a cloud-native SIEM (Security Information and Event Management) and SOAR (Security Orchestration, Automation, and Response) solution built on Azure. Unlike traditional SIEMs that require infrastructure management, Sentinel is fully serverless.

ğŸ’¡ Key Characteristics
  • Cloud-Native: No servers to manage - runs entirely in Azure
  • Pay-per-use: Cost based on data ingested (GB/day)
  • Built-in SOAR: Logic Apps for automated response
  • Microsoft Integration: Native connectors to M365, Defender, Azure
  • KQL Query Language: Same language used across Azure monitoring

Core Components

Log Analytics Workspace

The foundation of Sentinel - where all data is stored and queried:

  • Tables: Data organized by type (SecurityEvent, Syslog, SigninLogs, etc.)
  • Retention: Interactive (hot) + archive tiers
  • Query: KQL (Kusto Query Language) for all analysis
ğŸ’¡ Workspace = Your SIEM Database

Think of Log Analytics Workspace as your SIEM's database. Sentinel is the security layer on top that adds detection rules, incident management, and automation.

Data Connectors

How data gets into Sentinel:

Connector Type How It Works Examples
Native (Service-to-Service) One-click enable, API-based M365 Defender, Azure AD, Azure Activity
Azure Monitor Agent (AMA) Agent on VMs, uses Data Collection Rules Windows Events, Linux Syslog
Syslog/CEF Forwarder Linux forwarder receives and sends Firewalls, network devices
REST API/Custom Push data via Data Collector API or custom connector Custom apps, unsupported sources

Analytics Rules

Detection rules that create incidents when conditions are met:

Rule Type Description Use Case
Scheduled KQL query runs on schedule (every 5min to 14 days) Most detection rules
NRT (Near Real-Time) Runs every minute, limited KQL Critical detections needing fast response
Fusion ML-based correlation of multiple alerts Advanced multi-stage attack detection
Microsoft Security Creates incidents from other Defender products Integrate MDE, MDO, MCAS alerts

Playbooks (Logic Apps)

Automated response workflows triggered by incidents:

  • Enrich incidents with external data
  • Send notifications (Teams, email, ticketing)
  • Take containment actions (disable user, isolate host)
  • Integrate with third-party tools

Interview Preparation

ğŸ’¼ Interview Tip

Common Question: "Explain Microsoft Sentinel architecture."

Good Answer: "Sentinel is built on Azure Log Analytics Workspace, which provides the data storage and KQL query engine. Data flows in through data connectors - native connectors for Microsoft services, Azure Monitor Agent for VMs, syslog forwarders for network devices, and REST APIs for custom sources. On top of the workspace, Sentinel adds security features: Analytics Rules for detection, Incidents for alert management, Workbooks for visualization, and Playbooks using Logic Apps for automated response. The key architectural point is that it's fully serverless - you don't manage any infrastructure."

ğŸ’¼ Interview Tip

Common Question: "When would you choose Sentinel over Splunk?"

Good Answer: "Sentinel makes sense for Microsoft-heavy environments. If the organization is using M365, Azure, and Defender products, Sentinel's native integration means faster deployment and better correlation. It's also ideal for cloud-first organizations that don't want to manage SIEM infrastructure. The pay-per-use model works well for smaller or variable workloads. Splunk is stronger when you need more flexibility with non-Microsoft sources, have existing Splunk expertise, or need on-premises deployment for data residency requirements."
