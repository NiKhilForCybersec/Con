Compliance & Governance

SIEM platforms are often central to meeting compliance requirements. Understanding regulatory frameworks and how SIEM supports compliance is essential for security consultants.

Compliance Frameworks Overview

Framework Scope Key SIEM Requirements
PCI DSS Payment card data Log retention (1 year), daily log review, file integrity monitoring
HIPAA Healthcare data Audit controls, access monitoring, breach detection
SOX Financial reporting Access controls, change management, audit trails
GDPR EU personal data Breach detection, data access logging, 72hr notification
SOC 2 Service organizations Security monitoring, incident response, logical access
NIST CSF General framework Detect function: continuous monitoring, anomaly detection

PCI DSS SIEM Reports

SPL: Daily PCI Review Dashboard
// Failed access attempts to CDE
index=* tag=pci_cde (action=failure OR EventCode=4625)
| stats count by src_ip, user, dest
| where count > 5
| sort -count

// Admin account usage
index=* tag=pci_cde user IN ("admin", "root", "administrator")
| stats count, values(action) as actions by user, src_ip, dest

// System object changes
index=* tag=pci_cde (EventCode IN (4720, 4722, 4725, 4726, 4738, 4740))
| table _time, user, EventCode, Account_Name, action

KQL: HIPAA Access Monitoring
// Monitor access to ePHI systems
SigninLogs
| where TimeGenerated > ago(24h)
| where ResourceDisplayName has_any ("EHR", "Epic", "Cerner", "Patient")
| summarize 
    AccessCount = count(),
    UniqueUsers = dcount(UserPrincipalName),
    Locations = make_set(Location)
  by AppDisplayName
| order by AccessCount desc

SOX Controls

Sarbanes-Oxley focuses on financial reporting integrity:

Key SOX Monitoring Areas

  • Access Controls: Who has access to financial systems and data
  • Change Management: Changes to financial applications and databases
  • Segregation of Duties: No single user can process and approve transactions
  • Privileged Access: Admin access to financial systems
SPL: SOX Change Monitoring
// Changes to financial applications
index=* tag=sox_systems (action=modified OR action=created OR action=deleted)
| stats count by app, user, action, object_type
| sort -count

// Privileged access to financial systems
index=* tag=sox_systems user_type=admin
| stats count, earliest(_time) as first_access, latest(_time) as last_access 
  by user, system
| eval duration_hours = round((last_access - first_access) / 3600, 2)

GDPR Breach Detection

GDPR requires breach notification within 72 hours:

KQL: Potential Data Breach Indicators
// Large data access by single user
OfficeActivity
| where TimeGenerated > ago(24h)
| where Operation in ("FileDownloaded", "FileCopied", "FileMoved")
| summarize 
    FileCount = count(),
    TotalSize = sum(FileSize),
    FileTypes = make_set(FileExtension)
  by UserId
| where FileCount > 100 or TotalSize > 100000000
| extend Alert = "Potential data exfiltration"

// Unusual access patterns
SigninLogs
| where TimeGenerated > ago(24h)
| where ResultType == 0
| where RiskLevelDuringSignIn in ("medium", "high")
| project UserPrincipalName, Location, RiskLevelDuringSignIn, 
          RiskEventTypes_V2, AppDisplayName

| tstats count WHERE index=* by index, sourcetype, _time span=1d
| stats 
    latest(_time) as last_event,
    avg(count) as avg_daily_events,
    sparkline(count) as trend
  by index, sourcetype

| sort hours_since_last desc

Q: "How does SIEM support compliance requirements?"

A: "SIEM is central to meeting compliance requirements across several dimensions:

  1. Centralized logging: Most frameworks require audit trails. SIEM provides a tamper-evident, centralized repository for logs from all systemsâ€”which frameworks like PCI DSS explicitly require.
  2. Access monitoring: HIPAA, SOX, and PCI all require monitoring who accesses sensitive data. SIEM correlates authentication logs to show access patterns and anomalies.
  3. Daily review evidence: PCI DSS requires daily log review. SIEM dashboards and scheduled reports provide evidence that reviews occurred and what was found.
  4. Breach detection: GDPR's 72-hour notification requirement means you need to detect breaches quickly. SIEM detection rules identify potential breaches in near-real-time.
  5. Retention compliance: Different frameworks require different retention periods. SIEM with tiered storage can maintain hot, warm, and cold data to meet requirements cost-effectively.

When designing SIEM for compliance, I ensure log sources cover all in-scope systems, retention meets the longest requirement, and reports map directly to control requirements for easy audit evidence."
