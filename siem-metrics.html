Detection Metrics

True Positive Rate

Metric Formula Target
True Positive Rate True Positives / Total Alerts > 80%
False Positive Rate False Positives / Total Alerts < 20%
Detection Rate by Rule TPs per rule / Total rule fires Varies by rule
SPL: Detection Quality Metrics
index=notable
| eval disposition = case(
    status IN ("resolved_true_positive", "closed_true_positive"), "TP",
    status IN ("resolved_false_positive", "closed_false_positive"), "FP",
    status IN ("in_progress", "new"), "Pending",
    true(), "Other"
)
| stats count by rule_name, disposition
| xyseries rule_name, disposition, count
| fillnull value=0
| eval total = TP + FP + Pending
| eval tp_rate = round(TP / (TP + FP) * 100, 1)
| where total > 10  // Rules with sufficient volume
| sort -FP
| table rule_name, TP, FP, tp_rate, total

MITRE ATT&CK Coverage

SPL: Detection Coverage
| rest /services/saved/searches
| search is_scheduled=1 disabled=0
| rex field=description "(?i)(?:T\d{4}(?:\.\d{3})?)" max_match=10
| mvexpand technique
| stats dc(title) as rule_count, values(title) as rules by technique
| lookup mitre_attack technique OUTPUT tactic, technique_name
| table tactic, technique, technique_name, rule_count, rules
| sort tactic, technique

Response Time Metrics

Metric Definition Target
MTTD (Mean Time to Detect) Time from attack start to alert < 1 hour (critical), < 24 hours (high)
MTTA (Mean Time to Acknowledge) Time from alert to analyst review < 15 minutes (critical)
MTTI (Mean Time to Investigate) Time to complete initial investigation < 1 hour (critical)
MTTR (Mean Time to Respond) Time from alert to containment < 4 hours (critical)
MTTC (Mean Time to Close) Full lifecycle from alert to closure Varies by severity
KQL: Response Time Metrics
SecurityIncident
| where TimeGenerated > ago(30d)
| where Status == "Closed"
| extend 
    TimeToAcknowledge = datetime_diff('minute', FirstActivityTime, CreatedTime),
    TimeToClose = datetime_diff('hour', ClosedTime, CreatedTime)
| summarize 
    AvgTimeToAcknowledge = avg(TimeToAcknowledge),
    MedianTimeToAcknowledge = percentile(TimeToAcknowledge, 50),
    AvgTimeToClose = avg(TimeToClose),
    MedianTimeToClose = percentile(TimeToClose, 50)
  by Severity
| order by Severity

Operational Health Metrics

Data Ingestion Health

SPL: Ingestion Monitoring
| tstats count WHERE index=* by index, sourcetype, _time span=1h
| timechart span=1h sum(count) by sourcetype limit=20
| predict algorithm=LL sum(count) as predicted holdback=0

// Gap detection
| tstats latest(_time) as last_event WHERE index=* by sourcetype
| eval hours_since = round((now() - last_event) / 3600, 1)
| where hours_since > 2
| sort -hours_since

Log Source Coverage

SPL: Coverage Tracking
// Compare expected vs actual sources
| inputlookup expected_log_sources.csv
| join type=left source_name [
    | tstats count WHERE index=* by sourcetype
    | rename sourcetype as source_name
]
| eval status = if(isnull(count), "Missing", "Active")
| stats count by category, status
| xyseries category, status, count

Q: "What metrics do you use to measure SIEM effectiveness?"

A: "I focus on metrics that drive real security outcomes:

  1. True Positive Rate by rule: Not overallâ€”per rule. This identifies which rules need tuning. I target >80% TP rate; rules consistently below 50% get fixed or disabled.
  2. Mean Time to Detect (MTTD): For incidents we do catch, how quickly? I measure from attack start (from forensics) to first alert. This validates detection timing.
  3. MITRE ATT&CK coverage: What techniques do we have detections for? This shows gaps. I prioritize coverage for techniques used by threats relevant to our industry.
  4. Alert backlog: Are we keeping up? A growing backlog means we're either understaffed or generating too many alerts. Both need addressing.
  5. Detection gaps from incidents: For every incident, what should we have detected but didn't? This drives detection improvement.

I avoid vanity metrics like 'total alerts'â€”high volume with low TP rate isn't success, it's alert fatigue. Quality over quantity."
