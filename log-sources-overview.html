Log Sources Overview

Prioritization framework for security log collection

The Log Source Prioritization Framework

Not all logs are equal. A strategic approach prioritizes sources that provide the highest security value relative to their volume and cost.

ğŸ’¡ The 80/20 Rule of Log Sources

Approximately 80% of your detection capability comes from 20% of your log sources. Focus on getting the critical sources right before expanding coverage.

Tier 1: Critical (Day 1 Priority)

Source Why Critical Key Events
Identity Provider All authentication flows through here Logins, failures, MFA challenges, password changes
Firewall/Proxy Network boundary visibility Allow/deny, C2 indicators, data exfiltration
EDR/Endpoint Where attacks execute Process creation, file writes, network connections
Email Gateway #1 initial access vector Phishing, malware attachments, BEC

Tier 2: Important (Week 1-2)

  • Windows Security Events - Authentication, process auditing
  • DNS Logs - C2 detection, data exfiltration
  • VPN Logs - Remote access visibility
  • Cloud Provider Logs - AWS CloudTrail, Azure Activity

Tier 3: Valuable (Month 1)

  • Web Application Logs - Custom app attacks
  • Database Audit Logs - Data access patterns
  • Linux/Unix Logs - Server activity
  • SaaS Application Logs - Salesforce, Workday, etc.

ğŸ’¼ Interview Tip

Question: "How do you decide which log sources to onboard first?"

Answer: "I prioritize based on security value and detection coverage. Tier 1 includes identity (all auth flows here), perimeter (firewall/proxy for network visibility), endpoints (where attacks execute), and email (top initial access vector). These four source types give you visibility into 80% of attack scenarios. After that, I add Windows events, DNS, VPN, and cloud logs. The goal is quick wins first - sources that are easy to onboard and immediately enable high-value detections."
