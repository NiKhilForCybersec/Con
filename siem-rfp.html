RFP Evaluation Criteria

Category Weight Key Questions
Detection Capabilities 25% Out-of-box rules, custom detection, MITRE coverage
Data Collection 20% Native connectors, custom parsing, cloud support
Total Cost 20% License, infrastructure, professional services
Usability 15% Analyst workflow, investigation UI, query language
Integration 10% EDR, SOAR, TI, ticketing integration
Scalability 5% Performance at scale, multi-tenant, global
Vendor 5% Viability, support quality, roadmap

Key RFP Questions

Detection & Analytics
  • How many out-of-box detection rules are included?
  • What is the MITRE ATT&CK coverage?
  • How are custom detection rules created and managed?
  • What UEBA/ML capabilities are included vs add-on?
  • How does correlation work across data sources?
Data Collection
  • What log sources have native integrations?
  • How is custom parsing handled?
  • What is the maximum ingestion rate supported?
  • How is data normalized across sources?
  • What cloud platforms are natively supported?
Pricing & Licensing
  • What is the pricing model (GB, EPS, users)?
  • What features require additional licensing?
  • Are there overage charges? How are they calculated?
  • What commitment terms are available?
  • What is included in support tiers?

Demo Scenarios

Standardized Demo Script
  1. Data onboarding: Show ingesting a new log source (Windows Security, cloud logs)
  2. Rule creation: Build a custom detection rule from scratch
  3. Investigation: Walk through investigating a sample alert
  4. Threat hunting: Demonstrate ad-hoc query capabilities
  5. Dashboard: Create an executive dashboard
  6. Integration: Show EDR/SOAR/TI integration
  7. Scale: Demonstrate performance with volume

Common Pitfalls

Selection Mistakes to Avoid
  • Underestimating data volume: Actual ingestion often 2-3x estimates
  • Ignoring hidden costs: Professional services, training, infrastructure
  • Feature checklist focus: Demo workflow matters more than feature lists
  • Skipping POC: Production testing reveals real limitations
  • Not checking references: Talk to similar-sized customers
  • Short-term thinking: Consider 3-5 year trajectory
  • Ignoring ecosystem: Integration with existing tools matters

Q: "How would you approach SIEM selection?"

A: "I follow a structured process:

  1. Requirements first: Before looking at vendors, I document use cases, data sources, volumes, and compliance needs. This prevents being swayed by shiny features.
  2. Shortlist carefully: I narrow to 3-4 vendors based on deployment model fit (cloud/on-prem), existing ecosystem, and rough budget alignment.
  3. Standardized evaluation: Every vendor gets the same demo script and RFP questions. This enables apples-to-apples comparison.
  4. POC with real data: For major investments, I run a proof of concept with actual production logs. Brochure performance rarely matches reality.
  5. TCO, not license cost: I calculate full 3-year TCO including infrastructure, services, and internal effort. The cheapest license often isn't cheapest overall.
  6. Reference checks: I talk to similar organizations about real-world experience, especially support quality and hidden gotchas.

The goal is matching the platform to actual needs, not buying the most features."
