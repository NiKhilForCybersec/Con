Splunk Query Optimization

SPL: Before and After Optimization
// âŒ SLOW - Full scan with late filtering
index=* | search user=admin | stats count by host

// âœ… FAST - Early filtering, specific index
index=authentication user=admin | stats count by host

// âŒ SLOW - Wildcard at start
index=windows CommandLine=*mimikatz*

// âœ… FAST - Avoid leading wildcards
index=windows CommandLine=*mimikatz.exe 
| where match(CommandLine, "(?i)mimikatz")

// âŒ SLOW - Unbounded time range
index=firewall | stats count by src_ip

// âœ… FAST - Time bounded
index=firewall earliest=-24h | stats count by src_ip
SPL: Use Summary Indexing
// Create summary index for repeated queries
| tstats count where index=firewall by src_ip, dest_ip, _time span=1h
| collect index=firewall_summary

// Query summary instead of raw data
index=firewall_summary | stats sum(count) by src_ip, dest_ip
SPL: tstats for Indexed Fields
// Standard search - searches raw events
index=windows EventCode=4624 | stats count by user

// tstats - uses indexed metadata (10-100x faster)
| tstats count where index=windows EventCode=4624 by user

KQL Query Optimization

KQL: Optimization Techniques
// âŒ SLOW - Filtering after expansion
SecurityEvent
| extend AllData = strcat(EventData, Computer, Account)
| where AllData has "admin"

// âœ… FAST - Filter first, then extend
SecurityEvent
| where EventData has "admin" or Account has "admin"
| extend AllData = strcat(EventData, Computer, Account)

// âŒ SLOW - Contains is expensive
SecurityEvent | where ProcessName contains "power"

// âœ… FAST - Use has for substring, has_cs for case-sensitive
SecurityEvent | where ProcessName has "power"

// âŒ SLOW - Parse everything
SigninLogs | extend Parsed = parse_json(tostring(DeviceDetail))

// âœ… FAST - Filter first, then parse
SigninLogs 
| where ResultType != 0
| extend Parsed = parse_json(tostring(DeviceDetail))
KQL: Materialized Views
// Create materialized view for common aggregations
.create materialized-view DailySigninSummary on table SigninLogs
{
    SigninLogs
    | summarize 
        SuccessCount = countif(ResultType == 0),
        FailureCount = countif(ResultType != 0)
      by UserPrincipalName, bin(TimeGenerated, 1d)
}

// Query the materialized view (much faster)
DailySigninSummary
| where TimeGenerated > ago(7d)
| summarize TotalSuccess = sum(SuccessCount) by UserPrincipalName

maxHotBuckets = 10                # More concurrent writes
maxWarmDBCount = 100              # Control warm bucket count
frozenTimePeriodInSecs = 7776000  # 90 days retention

# Separate high-volume sources
[security_network]
# Network logs - different retention, different tuning

[security_endpoint]
# EDR data - may need different settings
Index Separation Strategy
  • By retention: Separate indexes for different retention periods
  • By volume: High-volume sources in dedicated indexes
  • By access pattern: Frequently searched vs rarely accessed
  • By compliance: PCI/HIPAA data in separate indexes

Detection Rule Optimization

SPL: Efficient Scheduled Search
// âŒ INEFFICIENT - Runs every 5 min, searches 24h
// Schedule: */5 * * * *
// Time range: -24h
index=authentication action=failure | stats count by user | where count > 100

// âœ… EFFICIENT - Runs every 5 min, searches only new data
// Schedule: */5 * * * *
// Time range: -10m (with buffer)
index=authentication action=failure earliest=-10m 
| stats count by user 
| where count > 10  // Adjusted threshold for shorter window
KQL: Analytics Rule Optimization
// âŒ INEFFICIENT - Joins two large tables
SecurityEvent
| join kind=inner (
    SigninLogs | where TimeGenerated > ago(24h)
) on Account

// âœ… EFFICIENT - Filter before join, use lookup
let SuspiciousUsers = SigninLogs
| where TimeGenerated > ago(1h)
| where ResultType != 0
| distinct UserPrincipalName;
SecurityEvent
| where TimeGenerated > ago(1h)
| where Account in (SuspiciousUsers)

Performance Monitoring

SPL: Search Performance Analysis
index=_audit action=search info=completed
| eval search_time = total_run_time
| eval scan_count = scan_count
| stats avg(search_time) as avg_time, 
        max(search_time) as max_time,
        avg(scan_count) as avg_scanned
  by user, search
| where avg_time > 60
| sort -avg_time
SPL: Indexing Queue Health
index=_internal sourcetype=splunkd component=Metrics group=queue
| eval queue_fill = current_size / max_size * 100
| stats avg(queue_fill) as avg_fill, max(queue_fill) as max_fill by name
| where avg_fill > 50
| sort -avg_fill

Q: "How do you optimize SIEM performance?"

A: "Performance optimization is multi-layered:

  1. Query optimization: Filter early, avoid wildcards at start, bound time ranges. Using tstats in Splunk or has vs contains in KQL makes huge differences.
  2. Index design: Separate indexes by volume and retention. High-volume sources shouldn't compete with security logs for resources.
  3. Summary/acceleration: For repeated queries, use data models, summary indexes, or materialized views. Search pre-aggregated data, not raw events.
  4. Detection tuning: Scheduled searches should only search new data, not re-scan history. Match search window to schedule frequency.
  5. Capacity planning: Monitor queue depths, search times, and disk I/O. Scale before hitting limits.
  6. Data reduction: Don't ingest what you don't need. Filter at collection, not at search time.

I always establish baseline metrics firstâ€”you can't improve what you don't measure."

Query Speed Tips

  • Filter early in query
  • Avoid leading wildcards
  • Bound time ranges
  • Use indexed fields
  • tstats > stats in Splunk
  • has > contains in KQL

Target Metrics

  • Dashboard load: < 10 seconds
  • Ad-hoc search: < 30 seconds
  • Correlation rule: < 60 seconds
  • Queue fill: < 50%

Red Flags

  • index=* (scan everything)
  • *term* (double wildcard)
  • No time bound
  • regex on large datasets
  • Large joins without filters
