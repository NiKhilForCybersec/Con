Best Practices

Indicator Lifecycle Management

  • Expiration: All IOCs should have expiration dates. IP indicators typically valid for days-weeks, domains for weeks-months.
  • Confidence Scoring: Not all indicators are equal. High-confidence indicators alert; low-confidence enrich.
  • Deduplication: Multiple feeds often share indicators. Dedupe to avoid false inflation.
  • Context: An IP alone is not useful. Include threat type, actor, campaign context.

Avoiding False Positives

  • Exclude CDN IPs, shared hosting ranges
  • Validate indicators against known-good lists
  • Use confidence thresholds (e.g., only alert on confidence > 70)
  • Correlate with other indicators before alerting
SPL: Confidence-Based Alerting
index=proxy
| lookup threat_intel_domain domain as dest_domain 
    OUTPUT threat_type, confidence, last_seen, source
| where isnotnull(threat_type)
| eval alert_level = case(
    confidence >= 90, "critical",
    confidence >= 70, "high",
    confidence >= 50, "medium",
    true(), "low"
)
| where alert_level IN ("critical", "high")
| table _time, src_ip, user, dest_domain, threat_type, confidence, alert_level

Q: "How do you operationalize threat intelligence in a SIEM?"

A: "I approach threat intel integration in layers:

  1. Technical indicators (IOCs): These go into lookup tables for real-time matching against network traffic, DNS queries, and file hashes. I set confidence thresholdsâ€”high-confidence indicators generate alerts, low-confidence just enrich events.
  2. Tactical intelligence (TTPs): I translate these into detection rules. If intel reports an actor using a specific PowerShell technique, I build a rule detecting that pattern, tagged with the actor name.
  3. Operational intelligence: Campaign information helps prioritize. If we know an actor is targeting our industry, I increase severity on related detections.
  4. Lifecycle management: IOCs have short lifespansâ€”attackers rotate infrastructure. I auto-expire indicators and track intel freshness to avoid stale data generating false positives.

The key is not just matching indicators but adding context. 'Hit on known-bad IP' is less useful than 'Hit on APT29 C2 infrastructure active in financial sector campaign.'"
