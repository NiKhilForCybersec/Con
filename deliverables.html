Project Deliverables

What you get from a SIEM migration or deployment project

ğŸ¯ Why Deliverables Matter

Documentation isn't just paperwork - it's how knowledge stays with your organization after the consultant leaves. Good deliverables mean your team can operate, troubleshoot, and improve the SIEM independently.

Deliverables by Project Phase

Phase 1: Discovery Deliverables

Document Purpose Contents
Discovery Report Summarize current state and findings Stakeholder interviews, current architecture, pain points, risks
Log Source Inventory Catalog all data sources Source name, type, volume (EPS/GB), owner, priority
Detection Content Audit Document existing rules Rule name, logic, severity, MITRE mapping, effectiveness
Gap Analysis Identify what's missing Missing log sources, detection gaps, integration needs

Phase 2: Design Deliverables

Document Purpose Contents
High-Level Design (HLD) Architecture overview for stakeholders Architecture diagrams, component selection, integration points, data flows
Low-Level Design (LLD) Technical implementation details Configurations, naming conventions, RBAC, network requirements
Migration Plan Phased migration roadmap Timeline, waves, dependencies, rollback strategy, success criteria
Test Plan Validation approach Test cases, acceptance criteria, parallel running plan

Phase 3-4: Build & Migrate Deliverables

Document Purpose Contents
Configuration Guide How the SIEM is configured Workspace settings, retention, RBAC roles, integrations
Data Onboarding Guide How to add new log sources Connector setup, parsing rules, field mappings, validation
Detection Rule Catalog All detection content Rule name, query, severity, MITRE technique, tuning notes
Query Translation Catalog Mapping old queries to new Original SPL/AQL â†’ New KQL/SPL, logic changes noted

Phase 5-6: Validate & Handover Deliverables

Document Purpose Contents
Validation Report Prove system works correctly Test results, coverage comparison, performance benchmarks
Operational Runbooks Day-to-day procedures Monitoring checks, troubleshooting, maintenance tasks
Training Materials Team skill development Platform training, query writing, investigation workflows
Handover Checklist Confirm project complete Sign-off items, open issues, support transition

Key Document Deep Dives

High-Level Design (HLD)

The HLD is what you show to management and architects. It answers "what are we building?" without getting into configuration details.

What's in a Good HLD:

  • Executive Summary - One page, business-focused
  • Architecture Diagram - Visual overview of all components
  • Component Selection - Why we chose Splunk Cloud vs Enterprise, etc.
  • Data Flow Diagram - How logs move from source to SIEM
  • Integration Points - Connections to SOAR, ticketing, threat intel
  • Security Considerations - Encryption, access control, data residency
  • Sizing Estimates - Expected volume, storage, licensing

Operational Runbooks

Runbooks are step-by-step procedures for common tasks. They allow anyone on the team to perform operations consistently.

Example Runbook Structure:

Runbook Template 
## Runbook: Daily Health Check

**Purpose:** Verify SIEM is operating correctly
**Frequency:** Daily (start of shift)
**Time Required:** 15 minutes
**Prerequisites:** Read access to monitoring dashboards

### Steps:

1. CHECK INGESTION STATUS
   - Navigate to: Monitoring > Data Ingestion
   - Verify: All sources show green status
   - Action if red: See Troubleshooting section

2. CHECK ALERT QUEUE
   - Navigate to: Incident Review
   - Verify: No critical alerts older than 4 hours
   - Action if backlog: Escalate to shift lead

3. CHECK SEARCH PERFORMANCE
   - Run: | tstats count where index=* by index
   - Expected: Query completes in < 30 seconds
   - Action if slow: Check indexer health

### Troubleshooting:

| Symptom | Possible Cause | Resolution |
|---------|---------------|------------|
| Source shows red | Forwarder down | Restart forwarder service |
| No data for 1+ hours | Network issue | Check firewall rules |
| Slow searches | Indexer overload | Contact Splunk support |

### Escalation:
If issues persist > 30 minutes, escalate to: siem-engineering@company.com

Documentation Best Practices

  • Keep it current - Documentation that's outdated is worse than no documentation
  • Version control - Use Git or document management system with versioning
  • Make it searchable - Use consistent naming, tags, and structure
  • Include examples - Real queries, real screenshots, real scenarios
  • Assign owners - Someone should be responsible for keeping each doc current
