Detection & Analysis Phase

This is where SIEM expertise matters most. Key activities:

Initial Scoping

Answer critical questions quickly:

  • What happened? Type of attack, techniques used
  • When did it start? Timeline of initial compromise
  • What's affected? Hosts, accounts, data
  • Is it ongoing? Active threat actor presence
  • How did they get in? Initial access vector

SIEM Investigation Queries

KQL: User Activity Timeline
// Build complete timeline for compromised user
let TargetUser = "compromised.user@domain.com";
let TimeWindow = 7d;

union
    (SigninLogs 
     | where UserPrincipalName =~ TargetUser
     | project TimeGenerated, Source = "SignIn", Activity = strcat(ResultType, " - ", Location)
    ),
    (AuditLogs
     | where InitiatedBy.user.userPrincipalName =~ TargetUser
     | project TimeGenerated, Source = "AuditLog", Activity = OperationName
    ),
    (SecurityEvent
     | where Account contains TargetUser
     | project TimeGenerated, Source = "WinSecurity", Activity = Activity
    ),
    (OfficeActivity
     | where UserId =~ TargetUser
     | project TimeGenerated, Source = "Office365", Activity = Operation
    )
| where TimeGenerated > ago(TimeWindow)
| order by TimeGenerated asc
KQL: Host Activity Timeline
// Build timeline for compromised host
let TargetHost = "WORKSTATION01";
let TimeWindow = 7d;

union
    (SecurityEvent
     | where Computer has TargetHost
     | where EventID in (4624, 4625, 4688, 4698, 4720, 7045)
     | project TimeGenerated, Source = "Security", Activity = Activity, Details = CommandLine
    ),
    (Event
     | where Computer has TargetHost
     | where Source == "Microsoft-Windows-Sysmon"
     | project TimeGenerated, Source = "Sysmon", Activity = tostring(EventID), Details = RenderedDescription
    ),
    (DeviceProcessEvents
     | where DeviceName has TargetHost
     | project TimeGenerated, Source = "MDE", Activity = ActionType, Details = ProcessCommandLine
    )
| where TimeGenerated > ago(TimeWindow)
| order by TimeGenerated asc
KQL: Lateral Movement Search
// Find where compromised account authenticated
let CompromisedAccount = "admin.user";
let CompromiseTime = datetime(2024-03-15T10:00:00Z);

SecurityEvent
| where TimeGenerated > CompromiseTime
| where EventID == 4624
| where Account contains CompromisedAccount
| summarize 
    FirstSeen = min(TimeGenerated),
    LastSeen = max(TimeGenerated),
    LogonCount = count()
  by Computer, LogonType, IpAddress
| order by FirstSeen asc

Evidence Collection

Preserve evidence for investigation and potential legal action:

Evidence Type Collection Method Priority
Volatile Memory Memory dump before shutdown Critical - collect first
SIEM Logs Export relevant timeframe High - already preserved
Disk Image Forensic imaging High - after memory
Network Captures PCAP from affected segments Medium - if available
Cloud Logs Export from cloud providers High - retention limits
EDR Data Export from EDR console High - detailed telemetry
âš ï¸ Chain of Custody

Document who collected what, when, and how. Hash all evidence files. Store securely with access logging. This is essential if the incident leads to legal action.

Common Incident Types

Ransomware Incident

Phase Actions
Detection File encryption alerts, ransom notes, EDR detection
Immediate Isolate affected systems, disable network shares, preserve evidence
Scope Identify all encrypted systems, determine initial access, find backup status
Contain Block C2, disable compromised accounts, segment network
Eradicate Remove persistence, rebuild from clean images
Recover Restore from backups, validate integrity, monitor closely

Business Email Compromise

Phase Actions
Detection Impossible travel, inbox rule creation, unusual email patterns
Immediate Reset password, revoke sessions, check mail forwarding rules
Scope Review sent items, check for data exfiltration, identify contacted parties
Contain Remove malicious inbox rules, block attacker IPs, enable MFA
Notify Alert recipients of fraudulent emails, contact vendors if financial
Recover Conduct awareness training, implement additional controls

Incident Severity Classification

Severity Criteria Response Notification
Critical (P1) Active attack, data breach, ransomware, critical system down All hands, war room, 24/7 Executive, legal, potentially regulatory
High (P2) Confirmed compromise, malware infection, credential theft Dedicated team, business hours+ Management, affected business units
Medium (P3) Potential compromise, policy violation, contained threat Normal SOC workflow, escalated Team leads
Low (P4) Reconnaissance, blocked attack, security event Normal SOC workflow Logged for trending

Post-Incident Activities

Lessons Learned Template

  • Incident summary: What happened, timeline, impact
  • Detection: How was it discovered? Could we detect earlier?
  • Response: What worked? What was difficult?
  • Root cause: Why did this happen?
  • Recommendations: What should we change?
  • Action items: Specific tasks with owners and deadlines

Detection Improvements

Every incident should improve detection:

  • Could we have detected this earlier with a new rule?
  • Were there alerts that should have been investigated?
  • What IOCs can we add to threat intel?
  • What log sources were missing that would have helped?

Q: "Walk me through how you would respond to a ransomware incident."

A: "I follow the NIST framework:

  1. Immediate containment: Isolate affected systems from the network to stop spread. Don't power them offâ€”preserve volatile memory for investigation.
  2. Scope the impact: Use the SIEM to identify all systems with encryption indicators. Check for lateral movementâ€”where did the attacker go before deploying ransomware?
  3. Determine initial access: Work backward from first encrypted system. Check email logs for phishing, VPN logs for compromised credentials, perimeter logs for exploitation.
  4. Eradicate persistence: The ransomware is just the final stage. Find and remove any backdoors, scheduled tasks, or other persistence before recovering.
  5. Recover: Rebuild from clean images and restore from backups. Validate backup integrity before trusting them.
  6. Post-incident: Document lessons learned. Implement detection for the initial access vector. Add new IOCs to threat intel.

Key point: Don't pay the ransom unless absolutely necessaryâ€”it doesn't guarantee recovery and funds criminal operations."
